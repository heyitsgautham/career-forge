## Scope of Changes

### Area A — GitHub Import Modal

**File:** `project/frontend/src/components/dashboard/projects-list.tsx`  
**Component:** `GithubImportModal`

#### A1 — Visibility Filter (Public / Private / Both)

**Current state:**  
`filteredRepos` only filters by `searchQuery` (text match). `is_private` flag already exists on every repo object returned by the backend — it's just never used for filtering in the UI.

**Target UX:**  
A 3-way segmented control sits below the search input:
```
[ All ]  [ Public ]  [ Private ]
```
Default: `All` (no change in behaviour for existing users).

**Implementation:**
- Add local state: `const [visibilityFilter, setVisibilityFilter] = useState<'all' | 'public' | 'private'>('all')`
- Extend `filteredRepos` memo to apply the filter after the text search:
  ```ts
  .filter(repo => {
    if (visibilityFilter === 'public')  return !repo.is_private;
    if (visibilityFilter === 'private') return repo.is_private;
    return true;
  })
  ```
- Render a 3-button toggle row (styled the same as the Select/URL import mode tabs already in the modal — keeps visual consistency)
- Show a count badge per segment: `Public (12)  Private (4)` — compute from the full `userRepos` list before filtering

**No backend changes required** — `is_private` is already returned by `GET /api/projects/github/user-repos`.

#### A2 — Previously Imported Projects Not Visible While Loading

**Root cause:**  
`importedRepoNames` is a `Set<string>` built from the current projects list query. If the projects query is mid-flight when the modal opens (e.g., user opens the modal immediately after page load), the set is empty — repos that are already imported don't show the "Imported" badge, and the user might accidentally select them again.

**Fix:**
- In the parent `ProjectsList` component, wait for the projects query to settle before allowing the modal to open OR show a loading indicator inside the modal while `projectsLoading` is true.
- Pass `projectsLoading` as a prop to `GithubImportModal`.
- While `projectsLoading === true`, render a subtle banner at the top of the repo list:
  ```
  ⏳ Loading your imported projects…
  ```
  Checkboxes remain usable, but the "Imported" badge logic is suspended — don't mark any repo as imported until the projects query resolves.
- Alternatively (simpler): block the modal open button while `projectsLoading` and show a spinner on the button label. Confirm with the user's preference — a non-blocking banner is preferred.

**Concrete change:**
```tsx
// In GithubImportModal props
projectsLoading: boolean;

// Inside the repo list section (above the repo list div):
{projectsLoading && (
  <div className="flex items-center gap-2 rounded-md bg-muted/40 px-3 py-2 text-xs text-muted-foreground">
    <RefreshCw className="h-3 w-3 animate-spin" />
    Loading your imported projects…
  </div>
)}
```

#### A3 — All Import Tasks Spawn Simultaneously

**Root cause:**  
`importMutation.mutate(fullNames)` fires a single `POST /api/projects/ingest/github` with all `full_names` in one payload. The backend iterates over them in a `for` loop with `asyncio.sleep(0.1)` between each. Since all repos are in a single request, the user cannot control concurrency — all Bedrock calls fan out inside the single request handler.

The **UI symptom** — "all three spawn immediately" — means the loading state (the overlay with repo names) appears for all selected repos at the exact same moment rather than a sequential progress view.

**Fix — Phase 1 (quick win, no backend change):**  
Show a sequential progress UI on the frontend side even though the backend processes them internally:
- Replace the flat `importingRepos: string[]` state with `importProgress: Array<{ name: string; status: 'queued' | 'processing' | 'done' | 'failed' }>`
- Animate through statuses every ~N seconds as a simulated progress indicator (the real status arrives only at the end via the mutation's `onSuccess`)

**Fix — Phase 2 (proper fix, backend change):**  
Add a `POST /api/projects/ingest/github/batch` endpoint that accepts repos one-by-one and streams progress via SSE (Server-Sent Events) or a polling status key per-import-job. Frontend subscribes to the SSE stream and updates each row individually.

> **Recommendation:** Ship Phase 1 immediately (30 min), keep Phase 2 in the backlog.

#### A4 — Can't Switch Tabs While Import is Running

**Root cause:**  
The `importMutation.isPending` state may be used somewhere (or the loading overlay covers the entire page) while the long-running import is in flight. Users expect to navigate away — the import should run in the background.

**Current state:**  
Looking at `handleStartImport`, the modal closes and `importingRepos` is set, which shows a loading banner inside the Projects tab. There may be a blocking overlay at the dashboard level.

**Fix:**
- Import progress should be shown as a **non-blocking toast notification** or a **persistent banner at the top of the Projects tab** — never a full-page overlay.
- If there is a `z-50 fixed inset-0` overlay triggered during import, remove it or restrict it to the Projects tab content area only.
- The tab navigation bar must remain fully interactive while an import is running. Switching away and back should retain the import progress state (the `importingRepos` state lives in `ProjectsList`, which is only mounted when the projects tab is active — the state will reset on tab switch. Fix: **lift `importingRepos` and `importMutation` state to the dashboard page level** so it persists across tab switches).

**Concrete change:**
- Move `importingRepos` state and `importMutation` from `ProjectsList` to `DashboardPage` (or a React context)
- Pass `onStartImport`, `importingRepos`, `importMutation.isPending` as props down to `ProjectsList` and `GithubImportModal`
- Add a persistent top banner that shows when `importMutation.isPending === true` regardless of active tab:
  ```
  🔄 Importing 3 repositories… (you can keep working)
  ```

#### A5 — Newly Imported Project Doesn't Appear with Existing Projects

**Root cause:**  
After `importMutation.onSuccess`, `queryClient.invalidateQueries({ queryKey: ['projects'] })` is called — but the backend ingestion is async and may not be done by the time the query re-fetches. The re-fetch returns the old list before Bedrock has finished embedding the project.

**Fix:**
- Add a short optimistic UI insertion after `onSuccess`: immediately add a placeholder card for each successfully imported repo (with a "Processing…" badge) to the projects list
- The placeholder uses data available at import time (full_name → repo name split, known from the selected repos)
- When the next background refetch completes (staleTime is already set low), the real data replaces the placeholder
- Alternatively, poll the projects query every 5s for 30s after a successful import, then stop

```ts
// In importMutation.onSuccess after toast:
queryClient.invalidateQueries({ queryKey: ['projects'] });
// Poll for 30s to catch delayed DB write
const pollId = setInterval(() => {
  queryClient.invalidateQueries({ queryKey: ['projects'] });
}, 5000);
setTimeout(() => clearInterval(pollId), 30_000);
```

### Area B — Dashboard Sidebar Scroll Isolation

**File:** `project/frontend/src/app/dashboard/page.tsx`

#### B1 — Left Navbar Scrolls With Page Content

**Current state:**  
The outer layout div uses `flex min-h-screen`, which lets the page grow taller than the viewport. The `<aside>` uses `relative flex flex-col`, so it stretches to match the full document height — causing it to scroll together with the main content area when the user scrolls down.

**Expected UX:**  
The left sidebar should remain fixed/sticky in the viewport at all times. Only the main content area (to the right of the sidebar) should be scrollable.

**Root cause:**  
`flex min-h-screen` on the outer container allows the document to grow beyond 100vh. `relative` on `<aside>` means it participates in normal flow and stretches. `overflow-y-auto` was on an inner content `<div>` rather than on `<main>`, so the sidebar had no scroll boundary to anchor against.

**Fix applied:**
1. Change outer container from `flex min-h-screen` → `flex h-screen overflow-hidden`  
   — locks the viewport; nothing outside 100vh, no document-level scroll
2. Change `<aside>` from `relative flex flex-col` → `sticky top-0 h-screen flex flex-col overflow-hidden`  
   — sidebar occupies exactly one viewport height and stays anchored
3. Change `<main>` to be the scroll container: add `overflow-y-auto`; remove `overflow-y-auto` from the inner content `<div>`  
   — main is now the scroll ancestor, so sticky elements inside it (e.g. sticky header) work correctly

```tsx
// Before
<div className="flex min-h-screen">
<aside className="... relative flex flex-col">
<main className="flex-1 flex flex-col min-w-0">
<div className="flex-1 overflow-y-auto p-6 animate-fade-in-up">

// After
<div className="flex h-screen overflow-hidden">
<aside className="... sticky top-0 h-screen flex flex-col overflow-hidden">
<main className="flex-1 min-w-0 flex flex-col overflow-y-auto">
<div className="flex-1 p-6 animate-fade-in-up">
```

## Risk Notes

- **A4 state lift**: Moving `importMutation` to dashboard page level is the cleanest fix but requires prop-drilling through `ProjectsList` → `GithubImportModal`. An alternative is a lightweight Zustand/context slice — evaluate based on how many intermediate components need to be updated.
- **A5 polling**: 5s × 6 = 6 extra DynamoDB scans per import. At low user volumes this is fine. If multi-user concurrency becomes a concern, switch to WebSocket push or SSE (Phase 2).
- **B1 sidebar**: `sticky top-0` inside a flex container requires the parent to be `h-screen overflow-hidden` — if the parent grows beyond the viewport, `sticky` has no boundary to stick against.
