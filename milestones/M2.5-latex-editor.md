## Overview

An in-browser LaTeX editor embedded in the Resumes tab. Users open any generated resume and get a full IDE-like editing experience:

- **Left panel** — Toggle between two views:
  - **Code view** — Monaco editor showing the raw `.tex` source (syntax highlighting, auto-save)
  - **Preview view** — Live PDF preview (compiled on-demand, `<iframe>` from S3 presigned URL)
- **Right panel** — AI chat powered by Claude (Bedrock), always visible; understands the current `.tex` and can rewrite sections, improve wording, fix errors, or apply targeted edits in one click

The route `/dashboard/resumes/[id]/edit` already exists in the file tree — this feature fills it in completely.

## Tech Decisions

| Concern | Choice | Reason |
|---|---|---|
| Code Editor | `@monaco-editor/react` | VS Code engine, LaTeX syntax via `monaco-editor`, already knows LaTeX |
| PDF Preview | `<iframe>` from S3 pre-signed URL | Simple, no extra dep; works with existing S3 upload pipeline |
| AI Streaming | FastAPI `StreamingResponse` + `EventSource` (SSE) | Bedrock Claude already set up; SSE simpler than WebSocket for this |
| Resizable panels | `react-resizable-panels` | Lightweight, works with Tailwind |
| Auto-save | Debounced `PUT /api/resumes/{id}/latex` (2s delay) | Prevent data loss; no extra backend arch needed |
| Left view toggle | Local `useState` `'code' \| 'preview'` | Switch between Monaco + PDF in left panel |
| AI chat placement | Permanent right panel (not a drawer) | Always accessible; no expand/collapse UX issues |

#### 1.3  `POST /api/resumes/{id}/compile` — recompile to PDF
```
POST /api/resumes/{id}/compile
Body: { "latex_content": "..." }   # optional override; defaults to stored content
Response: {
  "pdf_url": "https://...",       # pre-signed S3 URL, 1-hour TTL
  "status": "compiled" | "error",
  "error_message": "..." | null
}
```
- [ ] Add route in `app/api/routes/resumes.py`
- [ ] Call existing `latex_service.compile()` with the provided (or stored) `latex_content`
- [ ] On success: upload PDF to S3, update `resume.pdf_path`, return pre-signed URL
- [ ] On error: return `{ status: "error", error_message: "<LaTeX stderr>" }`
- [ ] Max compile timeout: 30s

#### 1.4  `POST /api/resumes/{id}/ai-edit` — streaming AI edit
```
POST /api/resumes/{id}/ai-edit
Body: {
  "message": "Make the summary section more concise",
  "latex_content": "..."    # current editor content
}
Response: SSE stream
  event: delta     data: {"text": "Here is the updated..."}  (partial)
  event: latex     data: {"latex": "\\documentclass..."}     (full updated .tex)
  event: done      data: {}
```
- [ ] Add SSE streaming route using FastAPI `StreamingResponse`
- [ ] System prompt: give Claude the full `.tex` as context, instruct it to:
  1. Respond conversationally explaining the change
  2. Emit a `<latex>` block with the full updated `.tex`
- [ ] Parse streaming chunks from Bedrock — forward text deltas as SSE `delta` events
- [ ] On stream end: parse `<latex>...</latex>` block from full response, emit final `latex` event
- [ ] Client receives it, shows "Apply Changes" button with the new `.tex` ready to paste in

## Open Questions / Decisions Needed

1. **Monaco vs CodeMirror** — Monaco is heavier (~5MB) but has much better UX. CodeMirror 6 is lighter. **Recommendation: Monaco** (VS Code feel, LaTeX extension available).
2. **Auto-compile toggle** — on by default with 3s debounce, or off by default? Compilation takes a few seconds so on-by-default may feel laggy. **Recommendation: off by default**, manual Compile button + Ctrl+Enter.
3. **PDF preview** — should we use `react-pdf` (renders PDF in canvas, no browser PDF viewer UI) or `<iframe>` (simpler, shows browser's native PDF viewer)? **Recommendation: `<iframe>`** for simplicity, browser PDF viewer is familiar.
4. **Chat history persistence** — store in-memory only (lost on page refresh) or persist to DB? **Recommendation: in-memory only** for now, add persistence in a future pass.
